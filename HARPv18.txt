v17 → v18 Changelog

This release closes every item on the v17 roadmap. v18 extends the v17 SOEM port with EoE/FoE field-service diagnostics, adds a TPM-sealed encryption layer to the MRAM driver (closing the gap between tamper-evident and tamper-resistant), extends the TLA⁺ scheduler model with a formal jitter proof for the EtherCAT DC sync working-counter gate under worst-case frame loss, replaces the single-TPM attestation root with a 2-of-3 TPM quorum, and adds a bounded, WCET-safe, int8-only on-device fine-tuning path for the gait classifier driven by fleet-analytics-aggregated outcome labels.

All v15/v16/v17 behavior is retained unless explicitly noted as [v18] below. No existing safety gate, proof obligation, or requirement ID was removed or renumbered — v18 only adds new SR-* entries.

v18 → v19 Roadmap (proposed)

With the v17 roadmap fully closed, candidate items for v19 — none yet implemented, listed for planning purposes only:

FoE-based signed firmware staging (not auto-apply) — revisit the v17/v18-excluded FoE module specifically for a staging-only field-update path: a technician could push a signed candidate image to a quarantined flash bank over EtherCAT, with secure_boot.rs's existing OTP-rooted verification gating whether it is ever promoted to the active bank — closing the gap between "diagnostics over EoE" (v18) and "full field provisioning" without ever trusting an unauthenticated network write path.

Quorum-aware MRAM key sealing — v18 deliberately kept MRAM-key sealing single-TPM (see security/tpm_attest.rs module docs) for traffic/latency reasons; investigate a Shamir-shared key reconstructable from any 2-of-3 TPMs ONLY at boot (amortizing the multi-TPM cost over one reconstruction rather than every ~10 Hz commit), closing the remaining single-point-of-failure in the storage confidentiality chain.

Gait recalibration delta persistence across power cycles — .gait_recalib_delta (v18) currently resets to zero on every cold boot (QSPI section is RAM-shadowed, not itself persisted across power loss in this revision); evaluate whether persisting the bounded delta is worth the added MRAM-style commit traffic, or whether per-boot re-accumulation from a longer fleet-analytics window is preferable.

EoE-driven HJ grid hot-reload — now that EoE diagnostics (v18) and the offline HJ grid generator (v17) both exist, investigate whether a re-surveyed deployment environment's updated hj_table.bin could be staged (read-only, Merkle-verified, NOT live-patched into the running .hj_table ITCM section) over the same field-service EoE channel, rather than requiring a full reflash for environment changes.

TLA⁺ model coverage for the costmap-edge HJ-veto path — v18's jitter model covers the EtherCAT/scheduler interaction; the comms/zenoh_transport.rs + cognitive/dreamer.rs costmap-edge ingestion path (v17) has no equivalent formal timing model yet — evaluate whether its 10 Hz polling cadence warrants the same rigor now that two other subsystems (scheduler, EtherCAT) have it.
